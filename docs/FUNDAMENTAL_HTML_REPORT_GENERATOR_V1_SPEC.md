# Fundamental HTML Report Generator v1 Design Spec

日期：2026-05-21

阶段：Design Spec only

本阶段只新增设计文档，不修改代码、不修改 dashboard、不修改 pipeline、不修改 classifier、不修改 connector、不修改 schema、不修改 tests。

参考文件说明：用户指定的 `docs/reference/个股研究-中国长城.html` 在当前仓库中未找到；实际可读参考文件为 `examples/个股研究-中国长城.html`。本规格以该文件作为视觉和结构参考，但不继承其交易、技术面或价格执行内容。

## 1. 产品定位

`Fundamental HTML Report Generator v1` 是一个面向用户阅读和分享的单股基本面 HTML 研报生成能力。

目标用户请求：

```text
帮我生成一份 002050 的基本面分析报告
```

目标输出：

```text
output/reports/fundamental_report_002050.html
```

它与 Dashboard v3 的边界不同：

| 能力 | Dashboard v3 | HTML Report Generator v1 |
| --- | --- | --- |
| 定位 | 交互式查看和审计面板 | 面向阅读、归档和分享的单股 HTML 研报 |
| 形态 | Streamlit 本地应用 | 完整自包含 HTML 文件 |
| 重点 | 审计 evidence pack、prompt、raw JSON、report stale/mismatch | 专业中文基本面研究叙事 |
| 数据关系 | 读取现有结果并展示 | 基于真实数据、evidence pack 和 AI structured report 渲染 |
| 用户感受 | 审计工具 | 专业研报 |
| 禁区 | 不做交易终端 | 不做交易终端 |

核心原则：

- 只做基本面研究。
- 使用 `fundamental_skill` 真实公共数据链路。
- 使用 evidence pack 限定模型分析边界。
- 模型必须基于 evidence pack 做解释性分析，不得只复述字段。
- 输出为专业中文 HTML 研报，不是 JSON 展示页。
- 不连接交易账户，不输出交易建议，不引入技术面。

## 2. 参考 HTML 模块拆解

参考 HTML 的可取之处是“研报阅读结构”和“视觉层次”，不是交易内容。v1 应保留专业视觉表达，删除交易/技术面模块。

### 2.1 应保留并改造成基本面版本的模块

| 参考模块 | v1 改造方向 |
| --- | --- |
| 顶部 sticky nav | 保留，导航到基本面模块，不出现“行情”“K线”“买卖时机” |
| Hero 区 | 保留，但改为公司身份、基本面状态、证据置信度、评分、风险等级、数据质量 |
| 核心结论卡 | 保留并增强为研究员式结论，必须包含支持证据、限制因素、跟踪指标 |
| 公司画像 | 保留，增加 strategy_type / sub_type 中文解释和业务锚点 |
| 近期动态 | 保留，限定为财报、公告、新闻、订单、政策和行业变化；新闻源失败必须说明 |
| 主营业务构成 | 保留，强调收入占比、毛利率、核心/拖累业务和增长弹性来源 |
| 关键财务指标 | 保留，扩展现金流、应收、存货、合同负债、capex、自由现金流 |
| 宏观与周期定位 | 保留，改为行业周期和政策/需求背景，不把政策热度当业绩兑现 |
| 产业链拆解 | 保留，增强商业模式、上下游、客户、供应链、壁垒与证据强弱 |
| 质量评分 | 保留为基本面评分和财务质量诊断，不做交易评分 |
| 业绩弹性 / 情景分析 | 保留为基本面情景，不输出股价区间 |
| 风险分析 | 保留，解释为什么重要，覆盖业务/财务/行业/政策/数据缺失风险 |
| 估值分析 | 保留为估值解释，不输出目标价，不做买卖判断 |
| 对标分析 | 保留，按财务、现金流、业务结构和估值综合对比 |
| 跟踪计划 | 保留为必须跟踪指标和后续基本面复核条件 |
| 页脚免责声明 | 保留并强化安全边界 |

### 2.2 必须删除的交易 / 技术面模块

必须明确删除：

- K线图。
- 行情涨跌作为核心主视图。
- 价格 hero。
- 涨跌幅 hero。
- 目标价。
- 买入 / 卖出 / 加仓 / 减仓 / 清仓。
- 仓位建议。
- 止损 / 止盈。
- 盈亏比。
- 买卖时机。
- 技术面分析。
- 短线交易计划。
- 操作建议。
- 账户动作。
- 技术指标、均线、成交量图形、图形形态。

参考 HTML 中以下内容不得照搬：

- “中期目标价”“三档目标价”“向上空间”“盈亏比量化”。
- “回调介入方案”“短线”“追高需谨慎”等交易执行表达。
- “仓位≤15%”“操作建议”。
- K线和 MA5/MA20/MA60。
- “估值与买卖时机”整段。
- “执行清单”中的任何买卖、止损、仓位语义。

### 2.3 可以保留但必须重命名或降级的模块

| 原模块 | v1 名称 | 降级/改造规则 |
| --- | --- | --- |
| 行情 | 顶部公司信息区 | 不展示涨跌作为核心判断，只显示身份、状态和数据质量 |
| 任务锁定与核心矛盾 | 核心命题与基本面矛盾 | 只讨论经营命题，不讨论交易任务 |
| 公司筛选与质量评分 | 财务质量诊断 / 基本面评分 | 不含技术面和交易维度 |
| 业绩弹性测算 | 业绩弹性 / 基本面情景分析 | 只分析收入、毛利率、费用、capex、订单对利润的影响 |
| 估值与买卖时机 | 估值解释 | 不输出目标价，不输出买卖时机 |
| 跟踪计划与综合结论 | 必须跟踪指标 / 后续基本面复核条件 | 不写执行清单、仓位、买卖点 |
| 逻辑破坏条件（止损信号） | 后续基本面复核条件 | 删除止损语义，改为重新评估基本面假设 |

## 3. Fundamental HTML Report v1 模块结构

HTML 报告必须是中文研报结构。每个模块均应包含自然语言分析和数据依据，不得只展示字段。

### 3.1 顶部公司信息区

字段：

- 股票代码。
- 公司名称。
- 行业框架 `strategy_type` 中文解释。
- `sub_type` 中文解释。
- 基本面状态。
- 证据置信度。
- 基本面评分。
- 风险等级。
- 数据质量状态。
- 报告生成时间。

分析要求：

- 解释 `confidence` 是证据置信度，不是看好程度。
- 解释基本面状态来自 deterministic pipeline 与 AI report 的综合呈现。
- 数据质量异常时在 Hero 区可见，但不喧宾夺主。

### 3.2 核心结论

要求：

- 必须是研究员式自然语言。
- 必须包含数据支撑。
- 必须指出核心支持证据、核心限制因素、核心跟踪指标。
- 不得只说 `status` / `confidence` / `strategy_type`。
- 不得输出交易建议。

建议结构：

- 一句话研究结论。
- 支持证据：2-4 条，引用 evidence。
- 限制因素：2-4 条，引用 evidence 或 missing evidence。
- 核心跟踪指标：按优先级列出。
- 置信度解释：说明强证据和弱证据。

### 3.3 公司画像

包括：

- 主营业务。
- 收入结构。
- 公司属性 / 实控人，如可得。
- 所处行业框架。
- `sub_type`。
- 核心业务锚点。

分析要求：

- 说明公司到底赚什么钱。
- 区分事实、proxy 和推断。
- 如果业务构成缺失，不得编造收入占比或新业务贡献。

### 3.4 近期基本面动态

包括：

- 最近财报变化。
- 重大公告或新闻，如可得。
- 订单、客户、产能、政策、行业变化。
- 如果 news endpoint 失败，必须说明新闻源不可用，不能编造。

分析要求：

- 区分“事件发生”和“业绩兑现”。
- 新闻只作为线索，不自动等同于订单或收入。
- 明确哪些动态有 source trace，哪些只是缺失。

### 3.5 主营业务构成

包括：

- 各业务收入占比。
- 毛利率，如可得。
- 核心业务。
- 拖累业务。
- 增长弹性来源。

分析要求：

- 说明收入占比和毛利率对利润结构的影响。
- 如果 segment 缺失，写明“主营构成数据缺失，无法判断业务结构分化”。
- 不得把主题概念写成已验证收入。

### 3.6 关键财务指标

包括：

- 收入。
- 收入增速。
- 净利润。
- 扣非利润，如可得。
- 毛利率。
- 净利率。
- ROE。
- 经营现金流。
- 应收账款。
- 存货。
- 合同负债。
- capex。
- 自由现金流。

分析要求：

- 表格展示数值，旁边给解释性摘要。
- 比例字段优先使用 `display_value`。
- 金额字段如果单位置信度低，应弱化判断。
- 合同负债只可作为订单可见度 proxy，不等于真实订单或 backlog。
- capex 只代表长期资产购建现金支出，不等于产能确定释放。

### 3.7 财务质量诊断

必须分析：

- 经营现金流 / 净利润。
- 经营现金流 / 营收。
- 应收账款压力。
- 存货压力。
- 合同负债含义。
- capex 压力。
- 自由现金流。
- 收入和利润质量是否健康。

必须给出：

```text
健康 / 一般 / 承压 / 无法判断
```

并解释原因。

分析要求：

- 现金流与利润背离时必须说明。
- 应收和存货缺失时不得判断回款质量和库存压力。
- 自由现金流可派生为 `operating_cashflow - capex`，但必须标记为 derived observation。

### 3.8 核心命题与基本面矛盾

参考 HTML 的“任务锁定与核心矛盾”，但改成纯基本面。

必须回答：

- 当前公司最核心的基本面命题是什么。
- 属于扭亏、成长兑现、周期修复、订单兑现、产能消化、现金流修复，还是其他命题。
- 最大矛盾是什么。
- 哪些证据支持该命题。
- 哪些证据尚不足。

### 3.9 宏观与行业周期定位

包括：

- 行业处于什么阶段。
- 政策、周期、需求背景。
- 不得把政策热度当成业绩兑现。
- 必须说明缺什么数据会影响判断。

分析要求：

- 政策、景气、需求只能作为外部背景。
- 必须落回公司自身收入、订单、毛利率、现金流等证据。
- 若行业专属运营指标缺失，应降低置信度。

### 3.10 产业链与商业模式拆解

包括：

- 公司在产业链的位置。
- 上游、下游、客户、供应链。
- 公司赚什么钱。
- 毛利率来自什么。
- 竞争壁垒是什么。
- 哪些壁垒尚未被数据证明。

分析要求：

- 壁垒必须用数据或公开披露支持。
- R&D ratio 只能代表研发强度，不等于技术壁垒已确认。
- 客户和订单缺失时，不得声称需求确定。

### 3.11 估值解释

要求：

- 不输出目标价。
- 不说买卖。
- 不只展示 PE / PB / PS。
- 必须解释估值高低是否有基本面支撑。
- 如果缺同行对比，必须写“同行估值基准缺失，无法判断行业分位”。
- 必须说明估值消化需要什么证据，例如利润增长、订单兑现、现金流改善。
- 不得给出目标价格。

分析要求：

- PE/PB/PS 只是估值背景，不是单独结论。
- 亏损或利润波动时，应说明 PE 可解释性下降。
- 估值解释必须连接利润质量、现金流、增长兑现和同行可比性。

### 3.12 业绩弹性 / 基本面情景分析

要求：

- 只做基本面情景，不做股价情景。
- 可设计乐观 / 基准 / 悲观三种基本面情景。
- 解释收入、毛利率、费用率、capex、订单变化如何影响利润。
- 不输出股价区间。
- 不输出买卖建议。

建议字段：

- 情景名称。
- 基本面假设。
- 关键变量。
- 对收入/毛利率/利润/现金流的影响。
- 证据强弱。
- 需要验证的数据。

### 3.13 对标分析

要求：

- 同行业公司对比。
- 可比公司缺失时写 `peer benchmark missing`。
- 对比维度包括收入规模、毛利率、净利率、ROE、PE/PB/PS、现金流质量、业务结构。
- 不得用单一 PE 判断贵或便宜。

分析要求：

- 对标必须说明可比性边界。
- 业务模型不同的公司不能简单横向套估值。
- 缺同行估值时，不得判断行业分位。

### 3.14 风险分析

包括：

- 业务风险。
- 财务风险。
- 行业风险。
- 政策风险。
- 数据缺失风险。
- 关键证据缺口。

要求：

- 风险必须解释为什么重要。
- 风险必须引用 evidence 或 missing evidence。
- `risk_flags` 为空不代表没有风险；如果高优先级 evidence 缺失很多，必须提示“风险尚未被充分识别或量化”。

### 3.15 必须跟踪指标

必须按优先级展示：

- `high` / `medium` / `low`。
- 当前状态。
- 为什么重要。
- 下一步需要验证的证据。

不能只列指标名。

建议表格列：

| 指标 | 优先级 | 当前状态 | 当前值 | 为什么重要 | 下一步需要验证的证据 |
| --- | --- | --- | --- | --- | --- |

### 3.16 后续基本面复核条件

替代原 HTML 的执行清单。

要求：

- 只写基本面复核条件。
- 不写买卖点。
- 不写止损。
- 不写仓位。

示例：

- 毛利率是否修复。
- 订单是否披露。
- 经营现金流是否改善。
- 应收账款是否下降。
- 客户集中度是否改善。
- capex 是否开始对收入或利润形成可验证贡献。

### 3.17 数据质量与无法判断事项

必须单独成区：

- 哪些数据可用。
- 哪些数据缺失。
- 哪些结论无法判断。
- 哪些字段只能作为 proxy。
- proxy 不等于事实。

要求：

- news endpoint 失败必须可见。
- source trace 缺失必须可见。
- 字段单位置信度低必须可见。
- stale / mismatch 报告不得作为可信正文。

### 3.18 安全边界说明

页脚和结尾必须写明：

- 仅供基本面研究。
- 不构成交易建议。
- 不含目标价。
- 不含买卖建议。
- 不含技术面判断。
- 不连接交易账户。

## 4. 每个模块的 AI 分析要求

每个模块不能只展示数据，必须回答：

1. 这个数据说明什么？
2. 为什么重要？
3. 对当前基本面结论有什么影响？
4. 证据强还是弱？
5. 缺什么数据会改变判断？

统一输出原则：

- 每个判断必须引用 evidence pack 中的字段、证据项或 missing evidence。
- 必须区分事实、proxy、推断。
- 必须说明数据缺口对置信度的影响。
- 必须避免把主题热度、政策表述、新闻标题当成业绩事实。
- 必须用研究员式自然语言，而不是字段摘要。

## 5. 行业框架专属增强

HTML Report v1 必须适配现有 `strategy_type`。每个框架需要专属分析问题、关键证据、常见误判、必须跟踪指标和不能高置信的条件。

### 5.1 `resource_swing`

- 核心分析问题：利润是否主要受商品价格、产量、成本曲线和周期位置驱动。
- 关键证据：商品价格、业务构成、产量/销量、单位成本、毛利率、经营现金流。
- 常见误判：把商品价格短期波动当成公司长期价值；把外部价格当成技术面信号。
- 必须跟踪指标：核心商品价格、产量、单位成本、库存、经营现金流、capex。
- 不能高置信的条件：缺商品价格、产量、成本、业务构成或现金流关键数据。

### 5.2 `resource_core`

- 核心分析问题：资源禀赋、成本曲线、现金流和分红能力是否支撑核心资源属性。
- 关键证据：储量/资源量、产量、成本、商品价格、多矿种结构、经营现金流、资产负债。
- 常见误判：把周期弹性股当作核心资源底仓；忽略矿山成本和再投资。
- 必须跟踪指标：核心资源价格、产量、成本、经营现金流、capex、债务压力、分红。
- 不能高置信的条件：缺储量/产量/成本/现金流或资源品价格 freshness 不足。

### 5.3 `right_trend_growth`

- 核心分析问题：高景气链条是否已经转化为公司订单、收入、利润和现金流。
- 关键证据：收入增速、毛利率、订单/客户、业务构成、capex、应收、估值。
- 常见误判：把 AI/算力/供应链叙事当作确定业绩；忽略客户 capex 周期。
- 必须跟踪指标：订单、客户验证、收入增速、毛利率、应收、经营现金流、估值消化。
- 不能高置信的条件：缺订单、客户、收入拆分或现金流质量数据。

### 5.4 `semiconductor_cycle`

- 核心分析问题：公司处于国产替代、资本开支周期、库存周期的哪个位置。
- 关键证据：收入增速、毛利率、库存、订单、R&D ratio、capex、客户验证、估值。
- 常见误判：把国产替代叙事等同于收入兑现；忽视库存和下游资本开支周期。
- 必须跟踪指标：订单、库存、毛利率、研发强度、客户验证、capex、经营现金流。
- 不能高置信的条件：缺库存、订单、客户、国产替代收入或业务构成。

### 5.5 `stable_growth`

- 核心分析问题：收入稳定性、现金流、ROE 和回款质量是否支撑稳健成长。
- 关键证据：收入、ROE、经营现金流、应收、合同负债、订单稳定性、资产负债。
- 常见误判：只看订单或政策投资，不看回款和现金转换。
- 必须跟踪指标：订单、经营现金流、应收账款、ROE、毛利率、资产负债率。
- 不能高置信的条件：缺现金流、应收、订单稳定性或 ROE 数据。

### 5.6 `advanced_manufacturing_growth`

- 核心分析问题：传统主业、新能源车、机器人/高端制造新业务是否形成可验证增长。
- 关键证据：业务构成、收入增速、毛利率、客户、订单、新业务收入占比、capex。
- 常见误判：把机器人布局、客户传闻、供应链叙事当成确定收入。
- 必须跟踪指标：新业务收入占比、订单、客户验证、毛利率、capex、经营现金流。
- 不能高置信的条件：缺新业务收入、订单、客户、业务构成或客户集中度。

### 5.7 `satellite_communication_infrastructure`

- 核心分析问题：在轨卫星、轨位/频段、转发器/带宽资源是否能稳定变现。
- 关键证据：在轨卫星数量、容量利用率、单位带宽价格、客户结构、合同期限、capex、折旧摊销、经营现金流。
- 常见误判：把商业航天主题、卫星制造或政策热度当成通信基础设施收入。
- 必须跟踪指标：容量利用率、客户集中度、合同期限、卫星剩余寿命、capex、折旧摊销、自由现金流。
- 不能高置信的条件：缺容量、利用率、客户、寿命、折旧摊销或合同结构。

### 5.8 `low_altitude_economy_infrastructure`

- 核心分析问题：公司是否真实从事低空基础设施或运营服务，而非泛主题概念。
- 关键证据：低空收入占比、运营小时、飞行架次、平台调度量、客户/项目、合同、验收。
- 常见误判：把无人机/eVTOL/机场/汽车零部件泛概念混入低空运营服务。
- 必须跟踪指标：运营小时、飞行架次、项目验收、合同金额、客户结构、回款。
- 不能高置信的条件：缺低空收入、运营量、项目合同、客户和验收数据。

### 5.9 `life_science_cxo_services`

- 核心分析问题：CRO/CDMO/CXO 服务是否具备订单可见度、客户质量和产能消化能力。
- 关键证据：CXO 收入占比、在手订单、新签订单、客户集中度、海外收入、毛利率、应收、capex、产能利用率。
- 常见误判：把自有药品管线、普通制药、医疗器械或 AI 制药软件混为 CXO。
- 必须跟踪指标：订单、客户结构、海外收入、产能利用率、毛利率、经营现金流、应收。
- 不能高置信的条件：缺订单、客户、海外暴露、产能利用率或项目进度。

### 5.10 `ai_datacenter_infrastructure`

- 核心分析问题：公司是否真实具备 IDC 运营、电力/UPS、冷却/液冷基础设施业务兑现。
- 关键证据：数据中心收入占比、订单、客户、交付周期、机柜数、MW 规模、上架率/利用率、PUE、液冷收入。
- 常见误判：把光模块、PCB、服务器、芯片、储能光伏、普通 HVAC 混入数据中心基础设施。
- 必须跟踪指标：数据中心收入、订单、客户、交付、利用率、PUE、液冷批量订单、经营现金流。
- 不能高置信的条件：缺收入拆分、订单、客户、交付、利用率或 PUE。

### 5.11 `theme_only`

- 核心分析问题：主题是否能被主营收入、订单、客户和利润兑现验证。
- 关键证据：主营业务相关度、业务构成、订单、客户、收入贡献、现金流。
- 常见误判：把新闻热度、政策词、概念标签当作基本面。
- 必须跟踪指标：主题相关收入、订单、客户、毛利率、现金流、公告兑现。
- 不能高置信的条件：只有新闻或概念词，缺业务和财务兑现证据。

### 5.12 `unknown`

- 核心分析问题：为什么无法稳定分类，最缺什么数据。
- 关键证据：基础信息、主营业务、业务构成、财务指标、行业描述。
- 常见误判：强行套用热门框架。
- 必须跟踪指标：主营业务、收入结构、行业分类、财务完整性、新闻/公告可用性。
- 不能高置信的条件：基础身份、业务构成或财务指标不足。

## 6. HTML 视觉规格

视觉基调参考 `examples/个股研究-中国长城.html` 的深色研报风格，但必须去交易化。

必须具备：

- 深色主题优先。
- 支持浅色主题切换。
- 顶部 sticky nav。
- 卡片式模块。
- Hero 区。
- 核心结论高亮卡。
- 表格统一样式。
- 标签 badge。
- 风险等级 badge。
- 评分条 / score bar。
- 产业链图谱可选。
- 情景分析卡片。
- 风险列表。
- must-track 表格。
- 页脚安全边界说明。

Hero 区应显示：

- 股票代码、公司名称。
- strategy_type / sub_type 中文解释。
- 基本面状态。
- 证据置信度。
- 基本面评分。
- 风险等级。
- 数据质量状态。
- 报告生成时间。

必须删除：

- 价格 hero。
- 涨跌幅 hero。
- K线图。
- 目标价。
- 盈亏比。
- 买卖时机。
- 技术面模块。

视觉风格建议：

- 深色背景：接近 `#0c0f15`。
- 卡片背景：接近 `#1a1c24`。
- 强调色：金色/琥珀色用于研究重点，红/橙仅用于风险，避免红绿涨跌语义。
- 浅色主题：保留阅读友好，不使用交易终端式强对比。
- 表格：统一 header、边框、hover，不用原始 dataframe 风格。
- 风险：按 high / medium / low 使用 badge，但不使用“止损信号”语义。
- score bar：只表示基本面评分或数据质量，不表示交易吸引力。

## 7. Prompt Builder v2 要求

Prompt Builder v2 应在现有 prompt-only 工作流基础上升级为 HTML report 所需 structured report JSON 生成约束。

核心约束：

- 必须解释，不得只复述数据。
- 每个判断必须引用 evidence。
- 缺数据必须说无法判断。
- 必须区分事实、proxy、推断。
- 不得输出交易动作。
- 不得输出目标价。
- 不得输出仓位。
- 不得输出技术面。
- 不得编造订单、客户、行业排名。
- 输出必须是中文。
- 结论必须像研究员分析，不像字段摘要。
- 必须生成 structured report JSON，可被 HTML renderer 渲染。

建议 structured report JSON 顶层结构：

```json
{
  "report_version": "fundamental_html_report.v1",
  "stock_code": "",
  "stock_name": "",
  "generated_at": "",
  "hero": {},
  "core_conclusion": {},
  "company_profile": {},
  "recent_fundamental_updates": {},
  "business_composition_analysis": {},
  "key_financial_metrics": {},
  "financial_quality_diagnosis": {},
  "core_thesis_and_contradictions": {},
  "macro_industry_cycle": {},
  "value_chain_business_model": {},
  "valuation_explanation": {},
  "fundamental_scenarios": [],
  "peer_benchmark": {},
  "risk_analysis": [],
  "must_track_indicators": [],
  "fundamental_review_conditions": [],
  "data_quality_and_unknowns": {},
  "safety_boundary": {}
}
```

每个分析段落建议包含：

```json
{
  "analysis": "",
  "evidence_refs": [],
  "evidence_strength": "strong | medium | weak | missing",
  "missing_data_that_could_change_view": []
}
```

Prompt 必须显式禁止：

- 交易动作词作为建议。
- 目标价格。
- 仓位比例。
- 技术指标。
- K线、均线、成交量、形态。
- 根据新闻或政策编造订单。
- 根据合同负债编造 backlog。
- 根据 capex 编造产能释放。
- 根据 R&D ratio 编造技术壁垒。

## 8. Report Generator v1 流程

用户请求：

```text
帮我生成一份 002050 的基本面分析报告
```

系统应执行：

1. run `real_stock_runner`。
2. build `evidence_pack`。
3. build prompt。
4. 使用 Codex / GPT-5.5 根据 prompt 生成 structured analysis report。
5. 保存 report JSON。
6. 渲染 HTML。
7. 输出：

```text
output/reports/fundamental_report_002050.html
```

建议文件产物：

```text
output/raw_002050.json
output/fundamental_002050.json
output/evidence_pack_002050.json
output/fundamental_report_prompt_002050.md
output/reports/fundamental_report_002050.json
output/reports/fundamental_report_002050.html
```

v1 决策：

- v1 可以在 Codex 对话中由模型生成 report JSON。
- 后续可选接 API mode。
- 当前不强制实现自动 API 调用。
- HTML renderer 应只消费 structured report JSON 与 evidence pack，不重新分析股票。
- 如果 structured report JSON 缺失或 safety 校验失败，不应生成可信 HTML 正文。

## 9. Evidence Pack v2 派生指标建议

### 9.1 v2 最小可实现

这些指标可以基于当前或已规划的 financial metrics 派生，缺字段时输出 missing：

- `operating_cashflow_to_net_profit`
- `operating_cashflow_to_revenue`
- `free_cashflow`
- `accounts_receivable_to_revenue`
- `inventory_to_revenue`
- `capex_to_revenue`
- `contract_liabilities_to_revenue`
- `gross_margin_trend`
- `net_profit_trend`
- `revenue_growth_quality`
- `valuation_peer_status`
- `order_visibility_status`
- `business_segment_quality`

解释规则：

- `free_cashflow = operating_cashflow - capex`，必须标记 derived。
- `contract_liabilities_to_revenue` 是订单可见度 proxy，不等于 backlog。
- `valuation_peer_status` 若缺 peer，值为 `peer benchmark missing`。
- `order_visibility_status` 若只有合同负债，值应为 `partial_proxy`。

### 9.2 future enhancement

这些指标可在后续版本增强，但 v1 不依赖：

- 多期收入、利润、毛利率趋势。
- 分业务毛利率趋势。
- 同行估值分位。
- 应收周转和存货周转。
- 客户集中度趋势。
- 研发效率。
- capex 与收入增长匹配度。
- 资产负债和利息覆盖。
- EBITDA / EV/EBITDA。

### 9.3 需要新数据源才可实现

这些指标需要新数据源或行业专属源，不能在 v1 编造：

- 真实订单 / backlog。
- 新签订单。
- 客户名单和客户收入占比。
- 产能利用率。
- 单位成本。
- 产销量。
- 数据中心机柜数、MW、PUE、上架率。
- 低空飞行架次、运营小时、平台调度量。
- CXO backlog、项目数量、产能利用率、海外收入细分。
- 卫星容量利用率、单位带宽价格、剩余寿命、转发器/带宽容量。
- 同行实时估值分位和可比公司自动匹配。

## 13. 是否建议进入 implementation

建议进入 `Fundamental HTML Report Generator v1 Implementation`，但 implementation 必须保持窄范围：

- 只新增 HTML report 生成能力。
- 不修改 deterministic pipeline 逻辑。
- 不修改 classifier。
- 不修改 connector。
- 不修改 scoring。
- 不修改 Dashboard v3。
- 不引入技术面。
- 不引入交易建议。
- 不接交易账户。

推荐下一阶段拆分：

1. `HTML Report Schema + Safety Spec`。
2. `Prompt Builder v2 implementation`。
3. `HTML Renderer v1 implementation`。
4. `Runner CLI implementation`。
5. `Golden sample for 002050`。
6. `Safety and rendering tests`。
